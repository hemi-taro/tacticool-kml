# Public Release Notes

Private development versions may be released publicly only after several internal versions.

For public GitHub Releases, write the release note as a cumulative summary from the previous public release to the new public release.

Do not list every private patch version in detail. Instead:

- Include major features added across skipped private versions
- Include important behavior changes
- Include user-visible fixes
- Group minor UI polish under "Improved UI" or similar wording
- Keep private implementation details in `CHANGELOG.md`

Example:

If the public release jumps from v1.3.3 to v1.4.2, the v1.4.2 GitHub Release note should summarize the main changes from v1.4.0, v1.4.1, and v1.4.2.
